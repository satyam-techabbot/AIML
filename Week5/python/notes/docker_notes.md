# 🐳 Docker – Notes

## 1. What is Docker?

**Docker** is a platform for developing, shipping, and running applications inside lightweight containers.

* Uses **containerization** (OS-level virtualization)
* Ensures **“works on my machine” → works everywhere**
* Faster than virtual machines

---

## 2. Key Concepts

### 🔹 Container

* Lightweight, standalone executable package
* Includes code, runtime, libraries, dependencies

### 🔹 Image

* Read-only template used to create containers
* Example: `nginx`, `ubuntu`

### 🔹 Docker Engine

* Core runtime that builds and runs containers

### 🔹 Dockerfile

* Script with instructions to build an image

### 🔹 Docker Hub

* Cloud registry for storing images
  (like GitHub for Docker images)

---

## 3. Docker Architecture

* **Client** → sends commands (`docker run`)
* **Daemon (dockerd)** → executes commands
* **Registry** → stores images

---

## 4. Virtual Machine vs Docker

| Feature     | VM          | Docker        |
| ----------- | ----------- | ------------- |
| Size        | Large (GBs) | Small (MBs)   |
| Boot time   | Slow        | Fast          |
| OS          | Full OS     | Shared kernel |
| Performance | Lower       | Near native   |

## 11. Volumes (Data Persistence)
Docker volumes are the preferred mechanism for persisting data generated by and used by Docker containers, ensuring that data remains intact even if the containers are stopped, removed, or replaced.

By default, data written inside a container is stored in its writable layer and is lost when the container is removed. 

Volumes solve this by storing data in a location on the host machine's filesystem that is managed by Docker, independently of the container's lifecycle.

```bash
docker volume create myvolume
docker run -v myvolume:/data ubuntu
```

## 14. Docker Compose

* Docker Compose is a **tool to define and manage multi-container Docker applications**.
* Instead of running multiple `docker run` commands, you describe all containers in a single **YAML file (`docker-compose.yaml`)**.
* Compose handles:

  * Network creation
  * Container orchestration
  * Volume management
  * Environment variables

---

### Why use Docker Compose?

* Run **multi-container apps** easily (web server + database + cache, etc.)
* Keep **configuration versioned** in a YAML file
* Simplify **development, testing, and deployment**
* Reproducible environments across machines

---

### Basic Concepts

| Concept                  | Description                                                |
| ------------------------ | ---------------------------------------------------------- |
| **Service**              | A container configuration (like `web` or `db`)             |
| **Project**              | The set of services defined in a `docker-compose.yml` file |
| **Network**              | Each project gets its own network (default)                |
| **Volume**               | Persistent storage for containers                          |
| **Environment variable** | Pass config settings to containers                         |

---

### Docker Compose File Structure

A `docker-compose.yml` has the following general structure:

```yaml
version: "3.9"  # Compose file version

services:
  service_name:
    image: image_name:tag   # Use pre-built image
    build: ./path_to_dockerfile  # Optional: build from Dockerfile
    ports:
      - "host_port:container_port"
    volumes:
      - host_path:container_path
    environment:
      - ENV_VAR=value
    depends_on:
      - other_service
```

### Key Sections:

* `version` – Compose file syntax version (3.x is most common now)
* `services` – Defines each container
* `build` – Build image from local Dockerfile
* `image` – Use an existing image
* `ports` – Map ports from container → host
* `volumes` – Map directories or create persistent volumes
* `environment` – Set environment variables
* `depends_on` – Set startup order between services

---

### Example: FastAPI + PostgreSQL

Project structure:

```
project/
  ├─ app/
  │   └─ main.py
  ├─ requirements.txt
  ├─ Dockerfile
  └─ docker-compose.yml
```

**Dockerfile (FastAPI)**

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**docker-compose.yml**

```yaml
version: "3.9"

services:
  web:
    build: .
    ports:
      - "8000:8000"
    volumes:
      - ./app:/app
    depends_on:
      - db

  db:
    image: postgres:15
    environment:
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: mypassword
      POSTGRES_DB: mydatabase
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

✅ Explanation:

* `web` service runs your FastAPI app
* `db` service runs PostgreSQL
* `depends_on` ensures `db` starts before `web`
* `volumes` persist PostgreSQL data across restarts

### Common Commands

| Command                               | Description                             |
| ------------------------------------- | --------------------------------------- |
| `docker-compose up`                   | Build & start all services (foreground) |
| `docker-compose up -d`                | Start services in background (detached) |
| `docker-compose down`                 | Stop and remove containers, networks    |
| `docker-compose ps`                   | List running services                   |
| `docker-compose logs`                 | Show logs of services                   |
| `docker-compose exec service_name sh` | Open shell in a running container       |
| `docker-compose build`                | Build or rebuild services               |
| `docker-compose restart`              | Restart services                        |

---

### Tips & Best Practices

* **Use `.env` files** for environment variables:

  ```env
  POSTGRES_USER=myuser
  POSTGRES_PASSWORD=mypassword
  ```

  And reference in `docker-compose.yml`:

  ```yaml
  environment:
    - POSTGRES_USER=${POSTGRES_USER}
    - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}
  ```

* **Use volumes for persistent data**, not inside the container

* **Avoid Alpine images for Python if using complex dependencies**

* **Use networks** to isolate services or allow cross-service communication

* **Tag your images** for production deployment


Used for multi-container apps.

### Example `docker-compose.yml`

```yaml
version: "3"
services:
  web:
    image: nginx
    ports:
      - "8080:80"
  db:
    image: mysql
    environment:
      MYSQL_ROOT_PASSWORD: root
```

### Run:

```bash
docker-compose up
```
